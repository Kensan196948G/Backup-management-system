## 1. デザイン哲学

### 1.1 基本原則

1. **シンプルさ** — 必要な情報のみを表示し、認知負荷を最小化する
2. **一貫性** — 全画面でデザインパターン・インタラクションを統一する
3. **フィードバック** — 操作結果を即座に視覚的・テキスト的に伝える
4. **効率性** — 最重要タスクを最小のクリック数で完了できる
5. **アクセシビリティ** — すべてのユーザーが利用できるインクルーシブデザイン

### 1.2 ターゲットユーザーペルソナ

**ペルソナ A: ITシステム管理者（主要ユーザー）**
- 経験: IT 管理 5 年以上
- 目標: バックアップ状況の即時把握・問題への迅速対応
- ペイン: 複数ツールを行き来したくない、情報が散在している

**ペルソナ B: バックアップオペレーター**
- 経験: IT 管理 1〜3 年
- 目標: 日常的なジョブ実行・確認作業の効率化
- ペイン: 操作手順が複雑で間違いやすい

## 5. コンポーネント設計原則

### 5.1 ボタン

| バリアント | 用途 | 視覚スタイル |
|---------|------|------------|
| Primary | 主要アクション（1 画面 1 つ） | 塗りつぶし（Brand Blue） |
| Secondary | 副次アクション | アウトライン |
| Ghost | 軽量アクション | テキストのみ |
| Danger | 破壊的操作（削除など） | 塗りつぶし（Error Red） |
| Icon | アイコンのみのボタン | ツールチップ必須 |

**状態:** Default / Hover / Active / Focus / Disabled / Loading

### 5.2 フォーム要素

- ラベルは常に入力の上部に表示（プレースホルダーはラベルの代替に使用しない）
- エラーメッセージは入力の下部に赤色テキストで表示
- 必須フィールドには「*」マークを付与
- フォーカスリングは 2px ソリッド Brand Blue

### 5.3 データテーブル

- ヘッダーはスティッキー（スクロール時に固定）
- 行ホバー時に背景色を薄く変化させる
- ソート可能列にはソートアイコンを表示
- 空状態には説明テキストとアクションボタンを表示

## 6. ナビゲーション設計

### 6.1 情報アーキテクチャ

```
ルート（/）
├── ダッシュボード（/dashboard）
├── ジョブ管理
│   ├── ジョブ一覧（/jobs）
│   ├── ジョブ詳細（/jobs/:id）
│   └── ジョブ作成（/jobs/new）
├── レポート（/reports）
├── 通知・アラート（/alerts）
├── 設定
│   ├── システム設定（/settings/system）
│   ├── ユーザー管理（/settings/users）
│   └── 通知設定（/settings/notifications）
└── ヘルプ（/help）
```

### 6.2 サイドバーナビゲーション

- 幅: 展開時 240px / 折りたたみ時 64px
- 折りたたみ状態ではアイコンのみ表示（ツールチップ付き）
- アクティブ状態は左ボーダー + 背景色で示す
- ユーザー情報はサイドバー最下部に表示

### 6.3 パンくずリスト

- 第 2 階層以降の画面に表示
- 最後の要素は現在のページ名（リンクなし）
